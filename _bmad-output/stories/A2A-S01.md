# A2A-S01 — A2A Bus (Agent-to-Agent Communication)

> **Epic** : A2A — Agent-to-Agent + Permissions (Noyau D)
> **Sprint** : Batch 11
> **Assignation** : Cofondateur
> **Effort** : L (8 SP, 5-7j)
> **Bloqué par** : CONT-S02 (Credential Proxy)
> **Débloque** : A2A-S02 (Permissions A2A), A2A-S03 (Audit A2A)
> **ADR** : ADR-006

---

## Contexte

Le A2A Bus permet la communication inter-agents au sein d'une company MnM. Quand un agent AI a besoin d'informations ou d'actions d'un autre agent, il émet une requête structurée via le bus. Le bus vérifie les permissions, détecte les cycles, applique le rate limiting (max 5 requêtes par chaîne), persiste les messages dans PostgreSQL, et notifie en temps réel via LiveEvents.

Pattern Saga avec TTL : chaque requête a un timeout configurable. Graph de communication avec détection de cycles. Chaîne maximale = 5 hops.

## Acceptance Criteria (Given/When/Then)

### AC1 — Envoi de message A2A
**Given** Agent A dans une company
**When** il envoie un message A2A de type "request" à Agent B
**Then** le message est persisté dans `a2a_messages` avec statut "pending", un LiveEvent `a2a.message_sent` est émis, et un audit event `a2a.message_sent` est créé

### AC2 — Réponse A2A
**Given** un message A2A "request" en statut "pending"
**When** Agent B envoie une "response"
**Then** le message original passe en "completed", la response est liée via `replyToId`, et un LiveEvent `a2a.message_responded` est émis

### AC3 — Détection de cycle
**Given** une chaîne A→B→C→A
**When** C tente d'envoyer un message à A
**Then** le message est rejeté avec erreur "CYCLE_DETECTED", un audit event `a2a.cycle_detected` severity "warning" est créé

### AC4 — Limite de chaîne (max 5 hops)
**Given** une chaîne A2A qui a atteint 5 requêtes
**When** la 6e requête est émise
**Then** elle est rejetée avec erreur "CHAIN_DEPTH_EXCEEDED", un audit event `a2a.chain_depth_exceeded` severity "warning" est créé

### AC5 — Rate limiting
**Given** un agent émetteur
**When** il dépasse 20 messages A2A par minute
**Then** les messages suivants sont rejetés avec erreur "RATE_LIMITED" et retryAfter en secondes

### AC6 — TTL expiration
**Given** un message A2A "request" avec TTL de 300s
**When** le TTL expire sans réponse
**Then** le message passe en statut "expired", un LiveEvent `a2a.message_expired` est émis

### AC7 — Notification de message
**Given** un message A2A envoyé
**When** la notification est créée
**Then** un LiveEvent est publié sur le companyId, contenant senderId, receiverId, messageType, et chainId

### AC8 — Erreur de message
**Given** un message A2A
**When** le traitement échoue (agent non trouvé, erreur interne)
**Then** un message de type "error" est créé avec le détail, et un audit event `a2a.message_error` est émis

### AC9 — Liste des messages A2A
**Given** un admin/manager
**When** il requête GET /companies/:companyId/a2a/messages
**Then** il reçoit la liste paginée des messages A2A filtrables par senderId, receiverId, messageType, status, chainId

### AC10 — Statistiques A2A
**Given** un admin
**When** il requête GET /companies/:companyId/a2a/stats
**Then** il reçoit totalMessages, pendingCount, completedCount, expiredCount, cyclesDetected, averageResponseTimeMs

### AC11 — Annulation de message
**Given** un message A2A en statut "pending"
**When** POST /companies/:companyId/a2a/messages/:id/cancel
**Then** le message passe en statut "cancelled", un audit event `a2a.message_cancelled` est créé

### AC12 — Nettoyage des messages expirés
**Given** le service A2A Bus
**When** le cleanup periodic s'exécute (toutes les 60s)
**Then** tous les messages pending dont le TTL est dépassé passent en "expired"

## Schema de données

### Table `a2a_messages` (nouvelle table — pas dans TECH-06)

| Colonne | Type | Contraintes | Description |
|---------|------|-------------|-------------|
| id | text | PK, uuid | Identifiant unique |
| companyId | text | FK companies, NOT NULL, INDEX | Tenant isolation |
| chainId | text | NOT NULL, INDEX | Identifiant de la chaîne de conversation |
| senderId | text | FK agents, NOT NULL, INDEX | Agent émetteur |
| receiverId | text | FK agents, NOT NULL, INDEX | Agent destinataire |
| replyToId | text | FK a2a_messages, NULLABLE | Lien vers le message auquel on répond |
| messageType | text | NOT NULL, CHECK | "request", "response", "notification", "error" |
| status | text | NOT NULL, CHECK, DEFAULT "pending" | "pending", "completed", "expired", "cancelled", "error" |
| content | jsonb | NOT NULL | Contenu structuré du message |
| metadata | jsonb | NULLABLE | Métadonnées additionnelles |
| chainDepth | integer | NOT NULL, DEFAULT 0 | Profondeur dans la chaîne (0-based) |
| ttlSeconds | integer | NOT NULL, DEFAULT 300 | Time-to-live en secondes |
| expiresAt | timestamp | NOT NULL | Date d'expiration calculée |
| respondedAt | timestamp | NULLABLE | Quand la réponse a été reçue |
| createdAt | timestamp | NOT NULL, DEFAULT now() | Date de création |

### Indexes
- `a2a_messages_company_idx` on (companyId)
- `a2a_messages_sender_idx` on (companyId, senderId)
- `a2a_messages_receiver_idx` on (companyId, receiverId)
- `a2a_messages_chain_idx` on (chainId)
- `a2a_messages_status_idx` on (companyId, status)
- `a2a_messages_expires_idx` on (status, expiresAt) — pour le cleanup

## Notes techniques

- **Pattern Saga** : chaque chaîne A2A a un `chainId` unique (UUID). Chaque message dans la chaîne incrémente `chainDepth`.
- **Détection de cycle** : avant d'envoyer, on récupère tous les agents de la chaîne (via chainId) et on vérifie que le receiverId n'est pas déjà dans la liste des senders.
- **Rate limiting** : in-memory Map (comme ChatWsManager), 20 messages/min par agentId. Cleanup periodic.
- **TTL cleanup** : setInterval toutes les 60s, UPDATE status = 'expired' WHERE status = 'pending' AND expiresAt < NOW().
- **LiveEvents** : même pattern que COMP-S01/DRIFT-S02 — publishLiveEvent avec companyId.
- **Audit** : auditService.emit() pour chaque action (send, respond, cancel, cycle_detected, chain_depth_exceeded, expired).
