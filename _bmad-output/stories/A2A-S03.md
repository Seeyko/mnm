# A2A-S03 — Audit A2A

> **Epic** : A2A — Agent-to-Agent + Permissions (Noyau D)
> **Sprint** : Batch 12
> **Assignation** : Cofondateur
> **Effort** : S (2 SP, 1-2j)
> **Bloque par** : A2A-S01 (A2A Bus), OBS-S01 (Table audit_events)
> **Debloque** : Aucun
> **ADR** : ADR-006, ADR-007

---

## Contexte

Le A2A Bus (A2A-S01) et les permissions A2A (A2A-S02) emettent deja des audit events via `auditService.emit()` pour certaines actions (message_sent, message_responded, message_cancelled, permission_denied, cycle_detected, chain_depth_exceeded, permission_rule_created/updated/deleted, default_policy_updated). Cependant, l'audit des communications A2A n'est pas complet :

1. **Messages expires** : le cleanup TTL expire les messages mais n'emet PAS d'audit event pour chaque message expire
2. **Audit enrichi dans le bus** : les metadonnees d'audit dans sendMessage/respondToMessage peuvent etre enrichies avec plus de contexte (content size, TTL restant, etc.)
3. **Consolidation des actions A2A** : toutes les actions A2A doivent suivre le pattern OBS-S02 `emitAudit()` dans les routes et `auditService.emit()` dans les services
4. **Audit de la resolution des permissions** : quand une permission est autorisee (pas seulement refusee), un audit event de type `a2a.permission_allowed` avec severity "info" doit etre emis pour la tracabilite complete

A2A-S03 consolide et complete l'audit de TOUTES les communications inter-agents pour garantir une tracabilite complete dans le systeme d'observabilite.

## Acceptance Criteria (Given/When/Then)

### AC1 — Audit on TTL expiration
**Given** un message A2A avec TTL expire
**When** le cleanup job `cleanupExpiredMessages()` l'expire
**Then** un audit event `a2a.message_expired` avec severity "warning" est emis pour CHAQUE message expire, incluant chainId, senderId, receiverId, ttlSeconds

### AC2 — Audit on permission allowed
**Given** le A2A Bus qui appelle `checkPermission()` dans `sendMessage()`
**When** la permission est autorisee
**Then** un audit event `a2a.permission_allowed` severity "info" est emis avec senderId, receiverId, senderRole, receiverRole, matchedRuleId, reason

### AC3 — Enriched metadata on message_sent
**Given** un message A2A envoye avec succes
**When** l'audit `a2a.message_sent` est emis dans le service
**Then** les metadonnees incluent contentSize (nombre de cles dans content), expiresAt, et le messageType

### AC4 — Enriched metadata on message_responded
**Given** une reponse A2A envoyee
**When** l'audit `a2a.message_responded` est emis
**Then** les metadonnees incluent responseTimeMs (temps entre la creation du message original et la reponse)

### AC5 — Audit actions consolidated
**Given** les fichiers a2a-bus.ts et a2a-permissions.ts
**When** on les examine
**Then** chaque operation mutante emet au moins un audit event avec les metadonnees appropriees

### AC6 — All A2A audit events include companyId
**Given** tous les audit.emit() dans le bus A2A et les permissions
**When** un audit event est emis
**Then** il contient toujours companyId (tenant isolation)

### AC7 — Severity levels follow pattern
**Given** les actions A2A
**When** un audit event est emis
**Then** severity = "info" pour les operations normales (sent, responded, cancelled, permission_allowed, rule CRUD), severity = "warning" pour les anomalies (expired, permission_denied, cycle_detected, chain_depth_exceeded)

### AC8 — Audit event for stats query
**Given** un admin qui requete les stats A2A
**When** GET /companies/:companyId/a2a/stats est appele
**Then** un audit event `a2a.stats_queried` severity "info" est emis (audit des acces en lecture aux donnees sensibles)

## Notes techniques

- **Pattern** : suit exactement le pattern OBS-S02 pour l'emission d'audit — `auditService.emit()` dans les services, `emitAudit()` dans les routes.
- **Non-blocking** : l'audit ne doit JAMAIS faire echouer l'operation business. Les erreurs d'audit sont loguees mais non propagees.
- **Performance** : dans cleanupExpiredMessages, l'audit est emis pour chaque message expire individuellement pour garantir la tracabilite complete, malgre le cout. Grouper par companyId pour les LiveEvents (deja fait).
- **Severite** : les operations normales utilisent "info", les anomalies "warning", les failles de securite "critical" (deja applique dans a2a-bus pour permission_denied).
