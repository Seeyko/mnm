> **Epic** : A2A — Agent-to-Agent + Permissions (Noyau D)
> **Sprint** : Batch 12
> **Assignation** : Cofondateur
> **Effort** : M (3 SP, 2-3j)
> **Bloque par** : A2A-S01 (A2A Bus)
> **Debloque** : A2A-S03 (Audit A2A)
> **ADR** : ADR-006

---

## Contexte

Le A2A Bus (A2A-S01) permet la communication inter-agents, mais le controle d'acces est actuellement base sur une seule permission generique `agents:create`. A2A-S02 introduit un systeme de permissions granulaires pour la communication A2A :

1. **Permission rules** : regles configurables par company definissant quels agents (par role/ID) peuvent communiquer avec quels agents
2. **Default policy** : politique par defaut (allow/deny) configurable par company
3. **Direction control** : regles unidirectionnelles ou bidirectionnelles
4. **Enforcement** : le A2A Bus verifie les permissions AVANT d'envoyer un message
5. **Admin CRUD** : API pour gerer les regles de permission A2A

Le systeme s'integre dans le service `a2a-bus.ts` existant via un nouveau service `a2a-permissions.ts` qui est appele avant l'envoi de chaque message.

## Acceptance Criteria (Given/When/Then)

### AC1 — Permission rule creation
**Given** un admin de la company
**When** il cree une regle de permission A2A (sourceAgentRole="engineer", targetAgentRole="qa", allowed=true)
**Then** la regle est persistee dans `a2a_permission_rules` et un audit event `a2a.permission_rule_created` est emis

### AC2 — Default deny policy enforcement
**Given** une company avec defaultPolicy="deny" et aucune regle matching
**When** Agent A (role="engineer") tente d'envoyer un message A2A a Agent B (role="designer")
**Then** le message est rejete avec erreur "A2A_PERMISSION_DENIED" et un audit event `a2a.permission_denied` severity "warning" est emis

### AC3 — Default allow policy
**Given** une company avec defaultPolicy="allow" et aucune regle matching
**When** Agent A envoie un message A2A a Agent B
**Then** le message est autorise (pas de blocage par les permissions)

### AC4 — Explicit allow rule override
**Given** une company avec defaultPolicy="deny" et une regle allow pour sourceRole="engineer" targetRole="qa"
**When** un agent engineer envoie un message a un agent qa
**Then** le message est autorise (la regle explicit override le default deny)

### AC5 — Explicit deny rule override
**Given** une company avec defaultPolicy="allow" et une regle deny pour sourceRole="engineer" targetRole="ceo"
**When** un agent engineer tente d'envoyer un message au CEO agent
**Then** le message est rejete avec "A2A_PERMISSION_DENIED"

### AC6 — Agent-specific rule (by ID)
**Given** une regle avec sourceAgentId=specificAgentId et targetAgentId=specificTargetId
**When** l'agent source envoie au target specifique
**Then** la regle ID-specific a priorite sur les regles role-based

### AC7 — Bidirectional rule
**Given** une regle avec bidirectional=true entre engineer et qa
**When** l'agent qa envoie un message a l'agent engineer
**Then** le message est autorise (la regle fonctionne dans les deux sens)

### AC8 — Permission rule listing
**Given** un admin
**When** il requete GET /companies/:companyId/a2a/permissions
**Then** il recoit la liste des regles de permission avec defaultPolicy, triees par priorite

### AC9 — Permission rule update
**Given** une regle existante
**When** l'admin la modifie (allowed: true -> false)
**Then** la regle est mise a jour et un audit event `a2a.permission_rule_updated` est emis

### AC10 — Permission rule deletion
**Given** une regle existante
**When** l'admin la supprime
**Then** la regle est supprimee et un audit event `a2a.permission_rule_deleted` est emis

### AC11 — Default policy update
**Given** la company avec defaultPolicy="allow"
**When** l'admin change le defaultPolicy a "deny"
**Then** le setting est mis a jour dans `companies` table et un audit event `a2a.default_policy_updated` est emis

### AC12 — Permission check on sendMessage integration
**Given** le A2A Bus
**When** sendMessage est appele
**Then** checkA2APermission() est appele AVANT l'insertion du message, et le resultat est audite

## Schema de donnees

### Table `a2a_permission_rules` (nouvelle)

| Colonne | Type | Contraintes | Description |
|---------|------|-------------|-------------|
| id | uuid | PK, defaultRandom | Identifiant unique |
| companyId | uuid | FK companies, NOT NULL, INDEX | Tenant isolation |
| sourceAgentId | uuid | FK agents, NULLABLE | Agent source specifique (prioritaire sur role) |
| sourceAgentRole | text | NULLABLE | Role source (engineer, qa, ceo, etc.) |
| targetAgentId | uuid | FK agents, NULLABLE | Agent target specifique |
| targetAgentRole | text | NULLABLE | Role target |
| allowed | boolean | NOT NULL, DEFAULT true | true=allow, false=deny |
| bidirectional | boolean | NOT NULL, DEFAULT false | Si true, la regle s'applique dans les deux sens |
| priority | integer | NOT NULL, DEFAULT 0 | Plus haut = plus prioritaire |
| description | text | NULLABLE | Description optionnelle |
| createdAt | timestamp | NOT NULL, DEFAULT now() | Date de creation |
| updatedAt | timestamp | NOT NULL, DEFAULT now() | Date de derniere modification |

### Indexes
- `a2a_perm_rules_company_idx` on (companyId)
- `a2a_perm_rules_source_role_idx` on (companyId, sourceAgentRole)
- `a2a_perm_rules_target_role_idx` on (companyId, targetAgentRole)
- `a2a_perm_rules_priority_idx` on (companyId, priority DESC)

### Modification table `companies`
- Ajout colonne `a2aDefaultPolicy` text DEFAULT 'allow' — politique par defaut ("allow" ou "deny")

## Notes techniques

- **Resolution de priorite** : quand plusieurs regles matchent, la regle avec le `priority` le plus eleve gagne. Les regles par agentId specifique ont une priorite implicite plus haute que les regles par role.
- **Lookup role** : pour resoudre le role d'un agent, on fait un SELECT sur la table `agents` avec le senderId/receiverId. Le champ `role` existe deja sur la table agents.
- **Default policy** : stocke sur la table `companies` comme `a2aDefaultPolicy` TEXT DEFAULT 'allow'. Cela evite une table separee.
- **Pattern** : meme pattern factory que a2a-bus.ts, credential-proxy-rules.ts, mount-allowlist.ts.
- **Audit** : utilise auditService.emit() pour chaque action CRUD + chaque permission denied.
